Overview

MachinePulse is a dashboard for monitoring industrial machines. It simulates live sensor readings for three machine types — a motor, a pump, and a conveyor — and turns those readings into a health score, a maintenance status, and an energy efficiency rating.

This matters because in real industrial settings, catching problems early — overheating, excess vibration, inefficient power draw — can prevent costly failures and downtime. Condition monitoring like this is a growing part of how modern equipment is managed, so I wanted to understand it by building a working version myself.

For this project, I am using MachinePulse as a way to build practical skills in software architecture, data handling, interface design, and cloud deployment.

Why I Chose This Project

I learn best through hands-on work, and I wanted a project that let me build the software and data side of an engineering system, to complement the hardware side I'm developing through my Battery Management System project.

This project also let me learn the full pipeline of building something real: writing the code locally, structuring it properly, version-controlling it with Git and GitHub, and deploying it so it's actually usable by anyone with the link, not just running on my own laptop.

What I Am Learning

  • Application structure: how to split a project into UI, logic, and database layers instead of one long script.
  • Health scoring: how to turn raw sensor values into a clear, interpretable status.
  • Data persistence: how to store and query readings over time using SQLite.
  • Version control: how Git and GitHub are used to track changes and back up a project properly.
  • Deployment: how to take a local Python app and host it live, so it runs independently of my own machine.

Design Considerations

A monitoring dashboard is only useful if its readings can be trusted. I put thought into how the health score is calculated — clamping it to a sensible 0–100 range, and weighting temperature, vibration, runtime, and power draw so no single factor unfairly dominates the result.

I also kept the code modular — separating the interface, the calculations, and the database — so each part can be tested and improved on its own without risking the rest of the app.

System Architecture

The system is built in layers, with the interface, the logic, and the data storage kept separate from each other.

  • UI layer: a Streamlit interface for selecting a machine, generating readings, and viewing live metrics.
  • Logic layer: functions that simulate sensor data and calculate health and power status.
  • Data layer: an SQLite database that stores every reading so trends can be tracked over time.
  • Visualization: line charts for temperature, vibration, power, and health score history.
  • Deployment: hosted on Streamlit Community Cloud, connected directly to the GitHub repository.

Predictive Maintenance Interest

One part of this project I am especially interested in extending is predictive maintenance — using historical trends to flag when a machine is likely to need service, instead of only reacting once a reading is already bad.

I have not implemented this yet, but since the history data is already being logged, it is a natural next step as the project develops.

Future Improvements

  • Add persistent cloud storage so history isn't lost when the app restarts.
  • Explore basic anomaly detection or predictive maintenance from historical trends.
  • Add alerting, such as a notification when a machine goes critical.
  • Add multi-user support with saved preferences per user.
